## 1. 简介：AI Agent 能为你做什么

RoboHire Agent 是你专属的 AI 招聘助理。你只需告诉它想招什么样的人,它就会自动:

- **搜索候选人** — 从你的简历库、公司共享简历池、以及第三方数据源中筛选
- **智能打分** — 用大模型分析每份简历,判断匹配度并给出 0-100 分
- **学习你的偏好** — 随着你点赞和拒绝,它越来越懂你想要的人
- **定时跑任务** — 你睡觉的时候也能自动扫一遍简历库找新人
- **给出修改建议** — 当你拒绝多个候选人后,它会分析原因并建议调整筛选标准

### 和传统的「关键词搜索」有什么不同?

| 传统搜索 | RoboHire Agent |
|---|---|
| 输入关键词,返回包含这些词的简历 | 理解你写的职位描述和候选人意向 |
| 每次搜索都要重新输入条件 | Agent 记住你的偏好,每次运行自动应用 |
| 不区分「必须有」和「最好有」 | 支持硬性条件(直接过滤)和软性标准(权重打分) |
| 一次只能搜一个岗位 | 每个 Agent 独立配置,并行运行多个招聘 |
| 不学习 | 你的每一次点赞/拒绝都在训练这个 Agent |

### 一张图看懂 Agent 的工作流程

```
[创建 Agent]
      ↓
  关联职位 + 写搜索条件 + (可选)硬性条件
      ↓
[运行 Agent] ← ─── 可以手动点,也可以定时触发
      ↓
[流式返回候选人] (匹配分 ≥ 60 才进入审阅)
      ↓
[审阅候选人] → 喜欢 / 忽略 / 审阅所有详细资料
      ↓
[Agent 学习你的偏好] → 生成 Ideal Candidate Profile (ICP)
      ↓
[下次运行更准] → 回到「运行 Agent」
```

## 2. 5 分钟快速上手

这一节帮你跑通第一个 Agent。详细功能在后面章节展开。

### 第 1 步 · 进入 Agent 页面

登录 RoboHire → 左侧导航 → 点击 **「Agents」**。

### 第 2 步 · 创建 Agent

点击右上角 **「+ Create New Agent」** (创建新 Agent)。填入:

| 字段 | 示例 | 说明 |
|---|---|---|
| **Agent Name** (名称) | `高级前端工程师 - 2026 Q2` | 给 Agent 起个好记的名字 |
| **Task Type** (任务类型) | `搜索候选人` | 绝大多数情况下选这个 |
| **Linked Job** (关联职位) | 从下拉框选一个你已创建的职位 | Agent 会按这个职位的 JD 来匹配 |
| **Candidate Sources** (候选人来源) | 勾选 `Instant Search` | 搜自己的简历库,最快最省成本 |
| **Search Criteria** (搜索条件) | `5 年以上前端,精通 React 和 TypeScript,有大型 SaaS 产品经验` | 用自然语言描述理想候选人 |
| **Hard Requirements** (硬性条件) | 可以先跳过,后面再加 | 见第 7 章 |
| **Schedule** (调度) | `Off` (关闭) | 第一次跑先用手动 |

点击 **「Create Agent」**。

### 第 3 步 · 运行 Agent

在 Agent 列表中点击你刚创建的 Agent,打开工作台。右上角点击 **「Run Now」** (立即运行)。

Agent 开始扫描简历,你会看到候选人陆续出现在 **「Results」** (结果) 标签页。匹配分 ≥ 60 的才会显示。

### 第 4 步 · 审阅

有两种审阅方式:

1. **列表视图** — 点击列表顶部的切换按钮 切换到列表模式,快速浏览所有候选人,用 👍 / 👎 按钮逐个决定
2. **审阅模式** (推荐) — 运行结束后,顶部会出现紫色卡片 **「I've found initial matches」**,点击 **「Review Profiles」** 进入全屏审阅模式,一次看一个,按 `A` 通过、`R` 拒绝

### 第 5 步 · 让 Agent 学习

审阅完一批后,**Results** 标签的顶部会显示运行摘要。摘要下方有两个按钮:

- **「Find more candidates」** — 寻找更多候选人(下次运行自动跳过你已看过的)
- **「Regenerate ideal profile」** — 让 Agent 根据你刚才的点赞/拒绝学习你的偏好

推荐先点「Regenerate ideal profile」,再点「Find more candidates」,这样第二次运行会更准。

**就这样!** 你已经掌握了 80% 的功能。下面是详细指南。

## 3. 创建你的第一个 Agent

### 3.1 打开创建对话框

在 **Agents** 页面右上角点击 **「+ Create New Agent」**。

### 3.2 填写表单

#### Agent Name · Agent 名称

给 Agent 起一个清晰的名字。好名字的例子:

- ✅ `高级 Python 后端 - 北京 - 2026`
- ✅ `ML 工程师 - 欧洲远程`
- ❌ `Agent 1` (一周后你不会记得这是什么)
- ❌ `工程师` (太泛)

#### Task Type · 任务类型

目前有两种:

- **Search Candidates** (搜索候选人) — 最常用。Agent 主动从简历库里找人。
- **Match Resumes** (匹配简历) — 你指定一批简历,Agent 只给这批简历打分。

#### Linked Job · 关联职位

选择你想为哪个职位招人。Agent 会读取这个职位的 JD (职位描述) 并把候选人跟它对比。

**重要**:管理员可以为任何招聘者的职位创建 Agent,普通招聘者只能选自己创建的职位。

#### Candidate Sources · 候选人来源

可以同时勾选多个(目前有三个):

| 来源 | 说明 | 何时用 |
|---|---|---|
| **Instant Search** (即时搜索) | 搜你自己上传的简历库 | 几乎所有情况,最快最省成本 |
| **Internal Repository** (内部资源库) | 搜公司共享的 MinIO 简历存档 | 想扩展到公司全量简历时 |
| **External Sources** (外部来源) | 通过管理员配置的第三方接口搜 (如 LinkedIn) | 需要管理员提前配置 |

**推荐**:第一次先只勾 Instant Search,跑通流程后再考虑扩展。

#### Search Criteria · 搜索条件

用自然语言描述你想要的候选人。这是 Agent 最重要的输入之一。

**好的写法**:
```
5 年以上后端开发经验,精通 Python 和 Go 其中至少一个,
有大型分布式系统或微服务架构经验,熟悉 Kubernetes 和 Docker,
有创业公司或快速成长期企业工作经验优先。
英文沟通能力良好(需要与海外团队协作)。
```

**不太好的写法**:
```
Python Go Kubernetes
```
这样只是关键词列表,Agent 不知道权重和语境。

#### Instructions · 指令 (可选)

对 Agent 的额外指示。例如:

```
请重点关注过去 3 年内的工作经历,早期经验仅作参考。
避免推荐频繁跳槽(3 年内跳槽 3 次以上)的候选人。
倾向于有完整项目交付案例的候选人,而不仅是技术栈匹配。
```

#### Hard Requirements · 硬性条件 (可选,但强烈推荐)

这是 Agent 最重要的一个新功能。详见 [第 7 章](#7-硬性条件-vs-软性标准)。

简单说:硬性条件是**必须满足**的过滤规则,不满足的候选人直接被排除,根本不进入 Agent 的打分流程。这能让你省 LLM 成本,也避免浪费时间审阅明显不合格的人。

常见例子:
- `工作经验 ≥ 5 年`
- `工作地点 = 北京 或 上海`
- `掌握语言包含 Python`
- `学历 ≥ 本科`

#### Schedule · 调度 (可选)

让 Agent 自动按计划运行。详见 [第 10 章](#10-定时任务--让-agent-自动工作)。

第一次建议设 **Off** (关闭),后面需要时再调。

### 3.3 点击「Create Agent」

创建成功后 Agent 会立即出现在列表中。如果你之前创建过其他 Agent 并进行过点赞/拒绝,新 Agent 会**自动继承**你的偏好(冷启动变温启动,详见第 12 章)。

## 4. 运行 Agent

### 4.1 打开 Agent 工作台

在 Agent 列表点击任意一个 Agent,打开它的工作台。工作台顶部有四个标签:

- **Results** (结果) — 候选人列表
- **Runs** (运行记录) — 历史运行
- **Activity** (活动日志) — Agent 做了什么的详细流水
- **Settings** (设置) — 修改 Agent 配置

### 4.2 点击「Run Now」

在工作台右上角点击 **「Run Now」**。Agent 会:

1. 加载你的「理想候选人档案」(ICP) 和硬性条件
2. 从指定来源拉取简历池
3. 应用硬性条件先过滤一波
4. 让大模型逐个打分
5. 保留 ≥ 60 分的候选人进入结果列表

结果会**实时**出现在 Results 标签页 — 不用等整个运行结束。

### 4.3 运行中能看到什么

在 Runs 标签页,正在运行的 Agent 会显示一个**动态卡片**:

- 🟣 紫色脉冲指示灯 + "RUNNING" 标签
- **Elapsed** — 已运行时长,每 2 秒更新一次
- **Last activity** — Agent 最新做的事情 (比如「正在评分张三」)
- **Matched / Scored / Errors** — 实时计数
- (仅管理员可见) **LLM 调用数、Token、成本**

### 4.4 取消运行

右上角的 **「Run Now」** 按钮在运行中会变成 **「Cancel」**。点击可以立即终止。

### 4.5 哪些候选人会被 Agent 跳过?

Agent 会**自动跳过**以下简历,避免重复浪费 LLM 成本:

1. 你这个 Agent 以前已经评估过的候选人(无论当时得分多少)
2. 被硬性条件过滤掉的候选人
3. 打分低于 60 的候选人

想重新评估已跳过的候选人? 去 Runs 标签页查看历史运行的结果。

## 5. 审阅候选人

Results 标签页提供**三种审阅方式**,你可以随时切换。

### 5.1 方式一 · 列表视图 (默认)

每行一个候选人,显示:
- 姓名 + 匹配分徽章
- 职位和地点
- "Why this candidate" 一句话评语 (Agent 生成)
- 👍 Like / 👎 Dislike 按钮

**快捷键**(鼠标悬停在列表时):
- `J` — 忽略当前候选人
- `K` — 喜欢当前候选人
- `L` — 打开详细资料

**过滤器**在列表顶部:
- **Pending** (待处理) — 还没决定
- **Liked** (已喜欢) — 已通过
- **Disliked** (已忽略) — 已拒绝
- **All** (全部)

### 5.2 方式二 · 审阅模式 (推荐新用户)

全屏一次看一个候选人,重点在深度审阅。进入方法:

- 运行结束后,顶部紫色卡片点 **「Review Profiles」**,或
- 列表视图顶部的**视图切换按钮**右侧那个,切换到审阅模式

布局:

```
┌─────────────────────────────────────────────┐
│  ← Review Profiles                          │
├──────────────────────────────┬──────────────┤
│  张三  [Full profile] [92]   │ + Add        │
│  北京, 中国    [🔗 领英 Github] │   Criteria  │
│                               ├──────────────┤
│  [经历] [教育] [技能]         │ ← Profile    │
│                               │    1/3    →  │
│  Senior Python Engineer       │              │
│  ByteDance, 2023 - Present    │ ┌──────────┐ │
│  • 负责推荐系统...            │ │ ✓ Approve│ │
│  • 主导性能优化...            │ │    [A]   │ │
│                               │ └──────────┘ │
│  Software Engineer            │ ┌──────────┐ │
│  字节跳动, 2021 - 2023        │ │ ✗ Reject │ │
│                               │ │    [R]   │ │
│                               │ └──────────┘ │
│                               │              │
│                               │ 仅用于校准   │
│                               │ Agent,不会   │
│                               │ 发送邮件。   │
└──────────────────────────────┴──────────────┘
```

**快捷键**:

| 按键 | 动作 |
|---|---|
| `A` | 通过 (Approve) |
| `R` | 拒绝 (Reject) |
| `←` | 上一位 |
| `→` | 下一位 |
| `Esc` | 返回列表 |

**左侧**显示候选人简历的完整内容,分三个子标签:
- **Experience** — 工作经历
- **Education** — 教育背景
- **Skills** — 技能树(按分类展示)

点击右上角的 **「Full profile」** 链接可以在新窗口打开完整简历。

**右侧**有导航箭头 + Approve / Reject 按钮 + 进度 (Profile 1/3)。

**关于「通过」和「拒绝」** — 重要说明:
- 这里的操作**不会发送邮件**给候选人
- 它只是告诉 Agent「我喜欢这个」或「我不喜欢这个」,用于学习
- 通过的候选人会进入「Liked」过滤器,后续你可以批量操作(邀请面试、发邮件等 — 这些功能在 Phase 6 开发中)

审阅完所有待处理候选人后,会显示绿色对勾 + 「All profiles reviewed」,点击返回列表。

### 5.3 方式三 · 卡片视图

(当前版本默认使用列表视图。卡片视图在后续迭代中提供。)

### 5.4 过滤和计数

无论哪种视图,顶部过滤器都显示实时计数:

```
[待处理 12]  [已喜欢 3]  [已忽略 5]  [全部 20]
```

### 5.5 滚动时的实时更新

如果 Agent 还在运行,新候选人会自动添加到列表顶部,**不需要刷新**页面。

## 6. 让 Agent 越用越聪明 · 学习机制

这是 RoboHire Agent 最强的功能。理解这一节能让你的 Agent 准确率大幅提升。

### 6.1 Agent 是如何学习的?

每次你审阅完一批候选人:

1. Agent 会分析你**喜欢了哪些人**和**为什么**(他们的共同点)
2. Agent 会分析你**拒绝了哪些人**和**为什么**(他们的共同特征)
3. Agent 会对比两组人的差异,总结出一份**「理想候选人档案」**(ICP)
4. 下次运行时,Agent 用这份档案来调整评分

这份 ICP 包含:

- **核心技能** (critical / high / medium 三档)
- **加分技能**
- **避免的特征** (anti-traits)
- **优先地区、行业、公司规模**
- **正向锚点** — Agent 引用的那些被你点赞的具体候选人作为「好的例子」
- **负向锚点** — 那些被你拒绝的候选人作为「不要找这种」的例子
- **置信度** — Agent 对这份档案的自信程度(0 到 1)

### 6.2 什么时候更新 ICP?

**需要你手动触发**。Agent 不会自动更新,因为更新涉及 LLM 调用有成本。

有两个入口:

**入口 A**:在 Results 标签页,运行结束后的紫色 delta 条会显示「Ideal profile updated · v3」,点击其中的 **「Regenerate ideal profile」**。

**入口 B**:进入 Settings 标签 → **Ideal Profile** 区域 → 点 **「Regenerate」**。

### 6.3 更新时会发生什么?

点击 Regenerate 后:

1. 弹出一个**对比模态框** (Diff Modal)
2. Agent 调用 LLM 分析你最近的点赞/拒绝(可能需要 5-15 秒)
3. 显示 **旧版本 (v3) → 新版本 (v4) 的差异**:
   - 添加的技能 (绿色)
   - 移除的技能 (红色)
   - 置信度变化 (e.g. `0.62 → 0.78 ↑`)
   - 叙述性总结的行级对比
4. 你可以选择:
   - **「Apply new version」** — 应用新版本 (默认)
   - **「Discard」** — 丢弃,恢复到旧版本

**重要**:新版本的 ICP 会**持久保存**,你可以随时查看历史版本并回退。

### 6.4 Agent 拒绝候选人给你建议

如果你拒绝了好几个候选人,运行摘要卡片上会出现一个额外按钮:

**「Get criteria suggestions」** (获取条件建议)

点击后 Agent 会:

1. 分析被你拒绝的候选人的共同特征
2. 生成 3-5 条可操作的修改建议,比如:
   - **Add** (新增条件): "考生必须有过独立带队经验" (因为你拒绝的都是纯个人贡献者)
   - **Modify** (修改条件): 把「5 年以上经验」改成「至少 7 年经验,其中 2 年管理经验」
   - **Remove** (移除条件): "你原来的'React' 要求可能过严 — 你拒绝的人都会 React,可能问题不在这"
3. 每条建议有 **「Apply」** 按钮,一键修改你的搜索条件

### 6.5 建议 vs. 自动修改

Agent **永远不会**自动修改你的搜索条件或硬性条件。所有修改都需要你点「Apply」确认。

## 7. 硬性条件 vs. 软性标准

这是 Agent 里最容易混淆但最关键的区别。

### 7.1 两者的区别

| 特性 | 软性标准 (Soft Criteria) | 硬性条件 (Hard Requirements) |
|---|---|---|
| 作用方式 | 影响**评分** | 直接**过滤** |
| 不满足会怎样? | 扣分,但仍会进入结果 | 根本不进入评估 |
| 例子 | 「有创业公司经验优先」 | 「工作地点必须在北京」 |
| 何时用 | 你希望的偏好 | 不满足就不考虑的硬性指标 |
| 会被 LLM 看到吗 | 会 | 不会(预先过滤掉了) |
| 费用影响 | 每个候选人都会被 LLM 评分 | 过滤掉的候选人不浪费 LLM 成本 |

### 7.2 什么时候用硬性条件?

用硬性条件的场景:

- ✅ 工作经验硬性要求 (e.g. `≥ 5 年`)
- ✅ 地点限制 (e.g. `只考虑北京或深圳`)
- ✅ 语言要求 (e.g. `必须掌握中文和英文`)
- ✅ 学历最低线 (e.g. `本科或以上`)
- ✅ 特定技能必备 (e.g. `必须掌握 Python`)
- ✅ 排除某些标签 (e.g. `排除派遣制员工`)

**不要用**硬性条件来表达「偏好」(那是软性标准的事)。如果你把「有 FAANG 经验」设为硬性条件,会直接过滤掉 90% 的人,大多数情况下不是你想要的。

### 7.3 怎么添加硬性条件?

在两个地方可以管理:

- **创建 Agent 时** — 在创建对话框中专门的「硬性条件」区域
- **编辑条件模态框** — 打开 Agent → Settings 标签 → Search Criteria 旁边的「Fine-tune criteria」链接 → 在条件模态框顶部的 **「硬性条件」** 区域

硬性条件编辑器是**琥珀色**的,带有「STRICT FILTER」标签,很容易识别。

每条规则由三部分组成:

| 部分 | 例子 |
|---|---|
| **Field** (字段) | 工作经验年数、地点、技能、语言、学历 等 |
| **Operator** (运算符) | `>=`, `<=`, `等于`, `包含任一`, `正则匹配` 等 |
| **Value** (值) | 数字、字符串、数组、正则 |

还有个**快速添加预设**:

- `至少 N 年经验`
- `位于...`
- `会某种语言`
- `持有某学位`

### 7.4 预跑检查 · 防止过度过滤

如果你设置的硬性条件会**过滤掉超过 90%** 的候选人,RoboHire 在实际运行 Agent 之前会弹一个**警告模态框**:

```
⚠️ 你的硬性条件会排除 397/420 位候选人

主要原因:
  · 地点 = 北京 (排除了 245 人)
  · 工作经验 ≥ 10 年 (排除了 132 人)
  · 语言 包含 日语 (排除了 89 人)

[← Edit]  [Run anyway]  [Cancel]
```

三个选择:
- **Edit** — 返回修改硬性条件
- **Run anyway** — 我就是想这么严格,继续
- **Cancel** — 完全取消本次运行

### 7.5 法律合规 · 不能用的字段

以下字段**绝对不能**作为硬性条件(基于反歧视法律):

- 年龄 (age)
- 性别 (gender)
- 种族 (race)
- 宗教 (religion)
- 国籍 (nationality)
- 婚姻状况 (marital)
- 怀孕 (pregnancy)

这些字段在下拉菜单里**根本不会出现**。即使你尝试用正则绕过,服务器端也会拒绝。

语言、地点、学历等**可以**使用,即使它们有时有相关性。

## 8. 寻找更多候选人

### 8.1 Find More Candidates 按钮

当一次运行结束,运行摘要卡片会有一个 **「Find more candidates」** 紫色主按钮。

点击它:

1. Agent 立即启动**新的一次运行**
2. 新运行**自动跳过**你这个 Agent 已经评估过的所有简历
3. 从简历库里**更深处**拉取新的候选人
4. 使用你**最新的 ICP**(如果之前更新过)
5. 结果实时流式返回到 Results 标签

### 8.2 和「Run Again」有什么区别?

(当前版本这两个功能合二为一。设计上我们计划分成两个按钮:)

- **Run Again** — 对同一批候选人重新打分,用于看最新的 ICP 如何重新排序
- **Find More** — 跳过已看过的,深度扩展

当前版本只有 **Find More**,这也是最常用的场景。

### 8.3 什么时候池子会空?

如果你的简历库只有 50 份简历,跑几次 Find More 之后可能就没有新的候选人可以评估了。这时候你有几个选择:

1. **扩大来源** — 勾选 Internal Repository 或 External Sources
2. **放宽硬性条件** — 可能你的条件过严
3. **调整搜索条件** — 可能你的描述太窄
4. **上传更多简历** — 让你的简历库更丰富

### 8.4 从已拒绝的候选人中重新考虑

已被你拒绝的候选人不会再次出现在新的运行中。如果你改变主意想重新考虑:

1. 去 Results 标签 → 切换过滤器到 **「Disliked」**
2. 点击候选人卡片上的 👍 按钮,把它们重新标记为 Liked

## 9. 运行摘要与活动日志

### 9.1 运行摘要 (Run Summary)

每次运行结束,Results 标签顶部会出现一个**摘要卡片**,包含:

**统计数据 (4 格网格)**:
- **Matched** — 最终进入结果的候选人数
- **Liked** — 你通过的
- **Pending** — 还没决定的
- **Errors** — 运行期间的错误数

**Top Candidates** — 按分排序的前 5 位候选人,每人显示:
- 排名序号(1-5)
- 姓名 + 匹配分
- 一句话简介
- Agent 生成的 reason

**Common Strengths** — 本次匹配中反复出现的优势 (emerald chips with `×N` 次数)

**Common Gaps** — 本次匹配中反复出现的短板 (amber chips with `×N` 次数)

**Actions** — Find more + Get suggestions

### 9.2 活动日志 (Activity Log)

点击 **Activity** 标签,你会看到 Agent 在本次运行(及历史)中做的**每一步**的流水:

```
20:32:15.234  run.started                 [AI 软件工程师 · system]
20:32:15.891  source.instant_search.hit   Queried 50 resumes
20:32:16.102  llm.call.started            Dispatching match for 张三  [1150ms]
20:32:17.253  llm.call.completed          Matched 张三 (score 85)     [1150ms]
20:32:17.260  match.scored                张三 scored 85
...
20:33:42.889  run.completed               8 matched, 42 scored
```

事件按类型**颜色编码**:
- 🟣 紫色 — LLM 调用 (llm.call.*)
- 🟢 绿色 — 成功匹配 (match.scored)
- 🔵 蓝色 — 源操作 (source.*.hit)
- 🟡 黄色 — 警告
- 🔴 红色 — 错误

每一行显示时间戳、事件类型、消息、Agent 名称、执行者。

Activity 标签是**实时的** — 事件在 Agent 运行时会不断滚入。

## 10. 定时任务 · 让 Agent 自动工作

### 10.1 为什么用定时任务?

如果你招的是长周期岗位(比如 6 个月招一个高级职位),你可以让 Agent 每天或每周自动扫描一次新上传的简历。你不用主动运行,它会在指定时间自动运行,结果累积在 Results 标签里等你审阅。

### 10.2 怎么设置?

创建或编辑 Agent 时,在 **Schedule** 区域选择一个预设:

| 预设 | Cron 表达式 | 何时运行 |
|---|---|---|
| **Off** | — | 不定时 |
| **Hourly** | `0 * * * *` | 每个整点 |
| **Daily** | `0 9 * * *` | 每天早上 9:00 |
| **Weekly** | `0 9 * * 1` | 每周一 9:00 |
| **Custom** | 自定义 | 你自己写 cron |

**Custom** 支持标准 5 字段 cron 表达式。例子:

| 场景 | Cron |
|---|---|
| 每个工作日早上 9 点 | `0 9 * * 1-5` |
| 每 30 分钟 | `*/30 * * * *` |
| 每周三 18:00 | `0 18 * * 3` |

### 10.3 定时运行的特殊行为

- Agent 会**自动跳过**已评估过的候选人(跟手动 Find More 一样)
- 如果服务器在计划时间不在线,重启后**会补跑**一次
- 定时运行失败会记录在 Activity 标签,但**不会**发邮件通知(后续迭代)
- 所有运行(包括定时的)都出现在 Runs 标签页

### 10.4 关掉定时

想暂停?编辑 Agent,把 Schedule 改回 **Off**,保存即可。Agent 会立即停止新的定时运行,但已在运行的会完成。

## 11. 设置、编辑与删除

### 11.1 Settings 标签

打开任意 Agent 的工作台 → 点击 **Settings** 标签。这里可以修改 Agent 的**所有**配置:

- 名称
- 任务类型
- 关联职位
- 候选人来源
- 搜索条件
- 指令
- 调度
- Ideal Profile(理想候选人档案)
- 硬性条件

### 11.2 Unsaved Changes 提示条

只要你修改任何字段,页面底部会出现**粘性保存条**:

```
🟡 You have unsaved changes                    [Discard] [Save changes]
```

- **Discard** — 放弃修改,恢复到服务器上的版本
- **Save changes** — 保存

### 11.3 Fine-tune Criteria 按钮

在 Search Criteria 区域的标签右侧有个 **「Fine-tune criteria」** 链接。点击它会打开完整的**条件模态框**,可以:

- 编辑硬性条件(顶部琥珀色区域)
- 管理 Most Important / Least Important 软性条件(带拖拽排序)
- 钉选条件 (pin) — 钉选的条件成为 LLM 必须考虑的「绝不妥协」项
- 保存为 Preset — 保存一套条件以便复用到其他 Agent
- 加载 Preset — 从已保存的预设中选一套

### 11.4 Danger Zone · 删除 Agent

Settings 标签最底部有一个**红色的 Danger Zone** 区域。

**Delete agent** 会:
- 删除 Agent 本身
- 删除它所有的运行记录
- 删除它所有的候选人
- 删除它的活动日志
- **不可恢复**

删除前会弹一个确认对话框。如果只是想暂停,请改用 Schedule = Off。

## 12. 跨 Agent 学习 · 你的招聘偏好档案

(Phase 7 新功能)

### 12.1 什么是招聘偏好档案?

RoboHire 不仅让每个 Agent 单独学习,还会把你**所有 Agent** 的学习结果**汇总**成一份属于你这个招聘者的**跨 Agent 偏好档案** (User Recruiter Profile)。

这份档案包含:
- 你最常喜欢的技能(来自所有 Agent 的点赞)
- 你最常拒绝的特征
- 你偏好的地区、行业、公司规模
- 你在多个 Agent 中反复设置的硬性条件

### 12.2 为什么有用?

**温启动** (warm start) 新 Agent。当你创建一个**新的** Agent 时,它不再从零开始 — 它会自动继承你的招聘偏好档案。第一次运行的准确率会比冷启动高得多。

例子:
- 你创建了一个**后端工程师**的 Agent,审阅了 20 个候选人,明显偏好有创业公司经验的人
- 你创建了一个**前端工程师**的 Agent,审阅了 15 个候选人,也偏好创业公司经验的人
- 现在你创建一个**全栈工程师**的新 Agent — Agent 立即知道你喜欢创业公司背景,第一次运行就把这类候选人排到前面

### 12.3 怎么关闭?

创建 Agent 时有一个**「Inherit from profile」** (从档案继承) 选项,默认开启。如果你想让某个 Agent 完全从零开始(比如转去招一个完全不同领域),可以在创建时取消勾选。

之后你可以随时在 Settings 标签手动切换。

### 12.4 档案什么时候更新?

每次你在**任何** Agent 上刷新 Ideal Profile 时,整体的招聘偏好档案也会跟着自动重建(有 60 秒节流防止频繁更新)。

### 12.5 重置档案

如果你换了一个公司,或者换了一个完全不同的招聘方向,可以完全重置档案:

- 去个人设置页 (开发中) → Recruiter Profile → **Reset**
- 重置后所有 Agent 会恢复到无继承状态

## 13. 常见问题

### Q1: 为什么我运行了 Agent 但没有任何结果?

可能原因:

1. **简历库是空的** — 去 Talent Hub 上传一些简历
2. **硬性条件过严** — 检查 Settings → 硬性条件,尝试放宽
3. **搜索条件和 JD 不匹配** — 大多数候选人得分 < 60 被过滤了
4. **所有简历都已被评估过** — Agent 自动跳过已评估候选人,你可能没有剩余可评估的简历。尝试上传新简历或切换到别的来源
5. **LLM 服务故障** — 查看 Activity 标签是否有红色 error 事件

### Q2: 点赞/拒绝会发送邮件给候选人吗?

**不会**。点赞/拒绝只是让 Agent 学习你的偏好,是纯内部的校准动作。候选人不会知道。

### Q3: 生成 Ideal Profile 大概要多久?

通常 5-15 秒,取决于你有多少点赞/拒绝的历史数据。你会看到一个加载动画,然后弹出 diff 对话框。

### Q4: Ideal Profile 可以回滚吗?

可以。每次生成都会创建一个新版本,旧版本保留在历史中。在 Ideal Profile Card 点击 **「View history」**,选择一个旧版本 → **「Revert」** 即可。

### Q5: Agent 能看到法律敏感字段(年龄等)吗?

**不能**。这类字段在硬性条件编辑器里**根本不可选**。即使它们出现在简历里,Agent 的匹配过程也不会用它们做任何过滤或评分决策。

### Q6: 我的搜索数据会被其他招聘者看到吗?

**默认不会**。每个 Agent 默认是私有的。管理员可以通过 Memory Manager 查看你的数据,但**每次访问都有审计记录**,你可以在 (即将开放) 的个人审计页查看谁看过你的数据。

### Q7: 定时任务会在我下班后跑吗?

会。定时任务是服务器端执行的,和你是否登录无关。如果你设置了每天 9:00 运行,即使你在睡觉它也会跑。

### Q8: 成本是怎么算的?

每次运行会调用 LLM 评估每个候选人(约 1-3 秒/人)。成本只对**管理员**可见,具体数字显示在运行摘要和活动日志里。单次运行通常在 $0.05 - $2 之间,取决于候选人数量和使用的模型。

### Q9: 我可以同时跑多个 Agent 吗?

可以。每个 Agent 独立运行,互不影响。但如果你的简历库不大,跑太多 Agent 会浪费 LLM 成本(因为它们可能评估相同的简历)。

### Q10: Agent 出错了怎么办?

1. 查看 **Activity** 标签是否有红色 error 事件
2. 查看 **Runs** 标签的运行状态,失败的运行会显示 `failed`
3. 如果是持续性问题,联系管理员

## 附录 A · 使用建议

### 建议 1: 从小处开始
第一次使用不要直接跑 500 个简历的大池子。从一个小职位 + 50 份简历开始,感受一下流程再扩大。

### 建议 2: 优先写好搜索条件
把 5 分钟花在写好 Search Criteria 上比事后拒绝 20 个候选人效率高得多。用自然语言写,写全,不要只堆关键词。

### 建议 3: 硬性条件要克制
把硬性条件限制在 3-5 条以内。你总能通过软性标准和 ICP 学习表达「偏好」,硬性条件只用于**绝对不能妥协**的点。

### 建议 4: 审阅 10-20 个后再生成 ICP
ICP 的质量随你审阅数量增加而提升。少于 5 次点赞时生成意义不大,10-20 次点赞后生成效果最好。

### 建议 5: 善用 Find More
不要在一个运行的结果里反复挣扎。评估一批 → 学习 → Find More → 新一批会更准。每次学习都是对 Agent 的训练。

### 建议 6: 新方向重置档案
如果你开始招一个完全不同领域的人(比如从后端工程师转招 HR),考虑在创建新 Agent 时取消「Inherit from profile」,避免跨领域的偏好互相干扰。

### 建议 7: 对敏感岗位用定时任务
高级岗位招聘周期长,设个每周一次的定时任务,让 Agent 帮你持续扫新上传的简历。你只需要每周审阅一次结果。
